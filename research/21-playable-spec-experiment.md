# Эксперимент: играбельная спека без изменения кода фреймворка

Дата: 13 июня 2026.

Вопрос: analint описывает и *проверяет* системы, но не *исполняет* их. Можно ли,
**не трогая код фреймворка вообще**, взять модель, написанную ради проверки, и
сыграть в неё как в текстовую игру — с текстом, выборами, здоровьем, инвентарём?
Все изменения — только в `examples/`.

Краткий ответ: **да, и удивительно чисто.** Одна и та же спека
`examples/sunless_crypt/spec.py` одновременно:

- проверяется: `analint examples/sunless_crypt/` → 9 сценариев + 5 reachability-
  запросов зелёные, 716 достижимых состояний;
- играется: `uv run python examples/play.py sunless_crypt` → полноценный
  проход от входа до концовки.

Исполнитель — `examples/play.py`, ~110 строк, живёт в папке примеров и
использует analint как библиотеку. Кода фреймворка не изменено ни строки.

---

## 1. Что выразилось в модели (и проверяется движком)

Вся **механика** легла на существующие примитивы:

| Игровое понятие | Примитив analint |
|---|---|
| Локации | `Room(StrEnum)` поле |
| Здоровье/стамина | `Field(5, ge=0, le=5, saturate=True)` — 0 = смерть, не уходит в минус |
| Инвентарь | bool-поля (`has_torch`, `has_sword`, `has_key`, `has_amulet`) |
| Выборы | `Action` с `pre`-гардами |
| Последствия | `effect` (перемещение, `Subtract(stamina, …)`, установка флагов) |
| Замки/ключи | `pre=[has_key == True]` на `unlock_vault` |
| Бой | `slay_guardian` (нужен меч) vs `fight_barehanded` (теряешь кровь) |
| Концовки | `Lifecycle` с `terminal=[ESCAPED, DEAD]` |

Бонус, которого нет у обычных игровых движков: поскольку игра — это
**проверяемая модель**, дизайн-свойства доказываются перебором всех 716
состояний:

- `Reachable(ESCAPED)` — игра проходима (с трассой-решением);
- `Reachable(DEAD)` — смерть достижима;
- `NoDeadEnd(goal=ended)` — **нет softlock**: из любого состояния есть путь к
  концовке;
- `AlwaysHolds(stamina >= 0)` — стамина не уходит в минус;
- `DeadActions()` — нет недостижимого контента.

То есть авторинг игры в analint бесплатно даёт верификацию «нет тупиков, весь
контент достижим», которую обычно ловят плейтестом вручную. Это сильный довод
для нарративного/игрового позиционирования (research/03).

## 2. Что в модель НЕ легло (честные границы)

### 2.1. Нарратив (текст, реплики) — вне модели

В модели нет понятия текста, и это правильно. Проза живёт рядом со спекой как
обычные питоновские данные: `describe(ctx)`, словарь `_ROOMS`, `INTRO`,
`ENDINGS`. Фреймворк их не видит (загрузчик собирает только analint-объекты).
Это подтверждает исходную интуицию: диалоги через DSL никто бы не описывал.

Важно: если бы текст всё же положить в `str`-поля модели, **explorer бы
подавился** (безграничный домен строк). Механика и проза тянут в разные стороны,
поэтому держать прозу снаружи — не костыль, а правильная граница.

### 2.2. Самого исполнителя в фреймворке нет

`play.py` пришлось собрать вручную, **переизобретая один «шаг»** из трёх
внутренних кусков:

- `evaluate(pred, ctx)` — выполнено ли предусловие;
- `_apply_effects(effect, ctx)` — следующее состояние;
- `Field.clamp` — насыщение границ (как делает движок).

Это ровно тот шов, который планируемый **transition kernel** должен выставить
наружу как переиспользуемый `step()`. То есть эксперимент независимо
**мотивирует kernel**: как только появится `step()`, раннер схлопнется в
тривиальный цикл, и не придётся вручную воспроизводить семантику (и рисковать
разойтись с движком — та самая проблема scenario↔explorer).

### 2.3. Форсированные переходы — нет понятия «автоматически vs по выбору»

Смерть при 0 стамины — не «выбор игрока». В модели все действия равны; нет
пометки «срабатывает само». Пришлось добавить в раннер хук `AUTO = {"succumb"}`
(игра объявляет форсированные id). Это смыкается с отложенной семантикой
`on=`/триггеров: настоящему исполнителю нужны автоматические/таймерные переходы.

### 2.4. Случайность — не выражается

В книгах-играх обычно кубики (бой, удача). analint моделирует недетерминизм как
ветвление/выбор, а не взвешенную случайность. Я обошёл это, сделав бой
детерминированным (меч vs голые руки). Раннер мог бы подмешать `random`, но
модель вероятность не выражает.

### 2.5. Ограниченность состояния — цена верифицируемости

Чтобы спека оставалась проверяемой (`PASS`, а не `INCONCLUSIVE`), состояние
держалось маленьким: стамина 0..5, несколько флагов, 6 комнат → 716 состояний.
Богатая игра (больше статов/предметов/комнат) пробьёт потолок ~10⁵ →
`INCONCLUSIVE`. Вывод: «верифицируемая игра» и «большая игра» — компромисс;
модель хороша для компактных связных дизайнов (головоломки, escape-room,
книги-игры), не для открытых миров.

---

## 3. Вердикт

Да, играбельную игру можно собрать с **нулём** изменений фреймворка: механика —
в модели (и проверяется), а проза, исполнитель, форсированные переходы и
случайность — снаружи. Эксперимент:

1. подтверждает направление «executable model» — модель не просто проверяется,
   на ней реально играешь;
2. независимо мотивирует два пункта роадмапа: **transition kernel** (как
   переиспользуемый `step()`, чтобы исполнитель не воспроизводил семантику
   руками) и **trigger/auto-семантику** (`on=`);
3. даёт неожиданный плюс: игра-как-модель бесплатно проверяется на отсутствие
   softlock и недостижимого контента.

Что НЕ нужно делать по итогам: тащить текст/диалоги/случайность в ядро. Граница
«механика в модели, нарратив снаружи» — здоровая и её стоит сохранить.

---

## 4. Артефакты

- `examples/sunless_crypt/spec.py` — модель (проверяемая) + нарратив-данные.
- `examples/play.py` — generic-исполнитель для любой analint-спеки:
  `uv run python examples/play.py <game> [--choices 1,3,2,...]`.
- В characterization-снапшоте `sunless_crypt` зафиксирован как обычный пример
  (716 состояний, win/death достижимы); 13 warnings — действия-перемещения без
  отдельных сценариев (на verdict не влияет).
