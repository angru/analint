# Мост спека↔код: пересмотр (якоря отклонены)

Статус: ревизия решения из research/08 §6.1 и ROADMAP v1.1 по итогам
обсуждения. Якоря `@implements("action-id")` в коде сервисов — **отклонены**.

## Почему якоря были плохой идеей

1. **Проверяют заявление, а не поведение.** Декоратор доказывает лишь,
   что кто-то *пометил* функцию. «У каждого действия есть якорь» — галочка.
   Самый опасный режим гниения: якорь на месте, поведение разъехалось —
   ложная уверенность хуже отсутствия проверки.
2. **Строковые ID гниют молча.** Переименование действия в спеке не знает
   о строках в N сервисных репозиториях. Обновлять их «не забудет» только
   дисциплина — то есть забудет.
3. **Социальная цена.** Мусор из строковых ссылок в боевом коде; команды
   будут сопротивляться, и правильно сделают.
4. **Категориальная ошибка.** Якоря решали задачу *локализации* («где в
   коде реализовано»), но трудная задача — *детекция* («код отстал от
   спеки»). Локализацию агент делает семантически и дёшево — каждый раз
   заново. Хранимый маппинг гниёт; перевычисляемый — нет.

## Принятая модель моста

### 1. Детекция дрейфа — пин версии спеки (без якорей)

В репозитории сервиса — один файл (`analint.pin`: repo спеки + commit).
CI сервиса сравнивает пин с актуальной спекой: спека ушла вперёд →
сигнал «нужна сессия сверки» + `analint diff <pinned>..HEAD` как рабочий
наряд. Обновление пина = осознанное действие в PR сверки.

### 2. Локализация — семантическая, силами агента

Агент получает дифф модели (v1.1 `analint diff`) и репозиторий сервиса —
дальше находит код по смыслу. Требование к экосистеме: репозитории
осмысленно нарезаны и названы (это и так стандарт приличной архитектуры).
Ничего не храним — нечему гнить.

### 3. Маршрутизация — маппинг на стороне спеки (системный слой)

Будущий слой системной аналитики (бывший v1.0-System из старого роадмапа):

```python
# в репо спеки, не в коде сервисов
billing = Service(
    repo="github.com/acme/billing",
    implements=[checkout, refund],     # ссылки на объекты, не строки!
)
```

Ключевое свойство: `implements` — питоновские ссылки. Переименование
действия ломается громко при загрузке спеки, в том же PR. Линтер получает
дешёвые проверки: каждое действие приписано какому-то сервису; дифф
сообщает «изменения задели billing и shipping». Это крупнозернистая
маршрутизация для агента, а не претензия на верификацию.

### 4. Поведенческая сверка (позже, по спросу) — конформанс через тесты

Если нужна жёсткая гарантия «код делает то, что спека обещает» — это
тесты, сгенерированные из сценариев спеки, живущие в тестовом каталоге
сервиса. Тесты — легитимное место для ссылок на требования (маркер на
тесте никого не смущает, в отличие от декоратора в доменной логике).
Боевой код остаётся чистым при любом исходе.

## Роль спеки — уточнение идентичности

Спека — **подсказка и верификатор для агента**, а не жёстко связанный
с кодом артефакт. Связь спека↔код держится на трёх китах: пин (детекция),
семантика агента (локализация), сервисный маппинг в спеке (маршрутизация).
Жёсткой синтаксической сцепки нет нигде — и это решение, а не упущение.

## Изменения в ROADMAP v1.1

Было: `@implements` + `analint coverage`.
Стало: `analint diff` (без изменений) + пин-файл и его проверка +
системный слой с `Service(implements=[...])` (перенесён ближе);
конформанс-тесты — отдельной строкой «по спросу».
