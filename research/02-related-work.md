# Родственные подходы: с чем сравнивать analint

Цель — не «выбрать конкурента», а понять, какие идеи каждая традиция даёт
и где на карте сидит analint.

## 1. Автоматическое планирование: STRIPS / PDDL / SAS+

**Самый близкий родственник.** PDDL-действие = `:precondition` +
`:effect` — структурно идентично `UseCase.rules` (PRECONDITION) +
`UseCase.effects`. analint фактически переизобрёл планировочный домен,
но без планировщика.

```pddl
(:action move-card
  :parameters (?c - card ?from ?to - column)
  :precondition (and (in ?c ?from) (not (archived ?c)))
  :effect (and (in ?c ?to) (not (in ?c ?from))))
```

Что есть у PDDL и нет у analint: **параметры действий**, **кванторы**,
условные эффекты, и главное — **поиск**: планировщик сам находит
последовательность действий из начального состояния к цели. Для сюжетов это
ровно вопрос «достижим ли финал».

Что есть у analint и нет у PDDL: человекочитаемость, типизированные сущности
с полями (PDDL — голые предикаты), сценарии как документация, нормальный
язык-носитель (Python вместо LISP-синтаксиса). PDDL никто не пишет ради
удовольствия — ниша «PDDL с эргономикой» реально пуста.

**Брать:** параметризованные действия, идею goal/reachability-запросов.

## 2. Model checking: TLA+ / Alloy / B-method

**TLA+** — состояние + действия-отношения + темпоральная логика; TLC
исчерпывающе перебирает состояния и находит контрпримеры. Это «золотой
стандарт» того движка, которого analint не хватает. Но порог входа высокий,
и нотация отпугивает — известная проблема adoption.

**Alloy** — реляционная логика + SAT-поиск контрпримеров в ограниченной
области (bounded scope: «проверь для ≤5 объектов каждого типа»). Идея
**small scope hypothesis** — большинство багов проявляется на маленьких
экземплярах — прямо применима: analint мог бы проверять «для ≤3 карточек и
≤2 досок» вместо синглтонов, и это уже качественный скачок.

**Брать:** у TLA+ — инварианты, проверяемые на *каждом* достижимом
состоянии (не только в сценариях); у Alloy — bounded scope и автогенерацию
контрпримеров вместо ручных `given`.

## 3. Statecharts (Harel) / SCXML

Расширение конечных автоматов: **иерархические состояния** (PAID — внутри
ACTIVE), **параллельные регионы** (статус оплаты × статус доставки),
guard-условия на переходах. Текущий `StateMachine` — плоский; для реальных
жизненных циклов иерархия нужна быстро. Это самое дешёвое из полезных
расширений: модель хорошо изучена, семантика стандартизирована (SCXML).

**Брать:** иерархию состояний, guards на переходах
(`Transition(A, B, when=Card.assignee_id != None)`).

## 4. Ceptre (Chris Martens) — линейная логика для игровых механик

Академически ближайшая работа именно к «верификации игр». Правила — это
производство/потребление ресурсов-фактов (линейная логика); Ceptre умеет
и исполнять (симуляция), и анализировать игровые механики и нарративные
структуры. Показала, что подход «правила + поиск» для сюжетов работает,
но осталась исследовательским прототипом с суровым синтаксисом.

**Брать:** трактовку фактов как ресурсов (создание/удаление сущностей —
естественная операция), режим «исполнить спецификацию как симуляцию».

## 5. Авторские инструменты нарратива: ink / Twine / Yarn Spinner

Это **языки написания** интерактивных историй, не верификации. ink (Inkle)
имеет переменные и условия, но проверка ограничена «компилируется ли».
Вопросы «достижима ли эта ветка», «можно ли застрять», «не противоречит ли
флаг X сцене Y» авторы проверяют плейтестами. Известная боль: в больших
играх (80 Days — ~750k слов) согласованность держится на ручной дисциплине.

**Вывод:** ниша «линтер для нарратива», который ест граф истории и отвечает
на вопросы достижимости/согласованности, — **свободна**. analint мог бы
быть верификационным слоем *поверх* таких инструментов (импорт из ink JSON —
отдельная интересная идея), а не конкурентом.

## 6. Machinations — диаграммы игровой экономики

Визуальный инструмент моделирования игровых ресурсов (источники, стоки,
конвертеры) с **симуляцией и Монте-Карло**: «через 100 шагов золото
инфлирует?». Это другой жанр — количественный баланс, а не логическая
согласованность. Полезен как ориентир, если analint пойдёт в сторону
симуляций: там ценность в стохастических прогонах и распределениях, а не
в доказательствах.

## 7. Puzzle dependency charts (Ron Gilbert) / lock-and-key графы

Индустриальная практика adventure-игр и Zelda-подобных данжей: DAG
«что нужно получить, чтобы открыть что». Это в точности `requires` +
events analint, только в индустрии это рисуют на доске. Формализация
lock-and-key — готовый, понятный гейм-дизайнерам словарь для нарративного
профиля analint.

## Карта

```
            ручные проверки ◄──────────────► исчерпывающий поиск
авторские   ink/Twine ──────── [ниша пуста] ─── Ceptre
системные   (Confluence/Word) ── analint ────── TLA+/Alloy
игровые     доски/чарты ───────── ▲ ─────────── PDDL-планировщики
                                  │
                        analint сегодня: язык уже
                        правее середины, движок — у левого края
```

Уникальная возможная позиция analint: **выразительность планировочного
домена + эргономика Python + движок уровня bounded model checking**.
Каждый элемент по отдельности существует; вместе — нет.
