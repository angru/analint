# Лоадер: кастомный обход файлов vs стандартный импорт Python

Сомнение: движок сам бегает по файлам директории (`importlib.util.
spec_from_file_location`, правки `sys.path`, `inspect.getmembers`). Не лучше
ли стандартный путь — точка входа с обычными импортами, которую исполняет
сам Python, плюс реестр объектов?

## Эмпирическая проверка: кастомный лоадер содержит баг

Эксперимент на taskboard (9 файлов, перекрёстные импорты):

```
>>> collected = collect_from_modules(modules)
>>> len(collected["entities"])
14                                  # должно быть 7!
>>> duplicate class names:
{'Board': 2, 'Card': 2, 'Column': 2, 'Comment': 2,
 'Membership': 2, 'Notification': 2, 'User': 2}
>>> [c.__module__ for c in card_classes]
['_analint_entities', 'examples.taskboard.entities']   # разные объекты
```

**Классическая проблема двойного импорта.** Каждый файл загружается дважды:

1. лоадер грузит `entities.py` напрямую как модуль `_analint_entities`;
2. `rules.py` делает `from examples.taskboard.entities import Card` —
   Python загружает **тот же файл ещё раз**, уже как пакетный модуль,
   потому что в `sys.modules` он лежит под другим именем.

Получаются два независимых класса `Card` с разными identity. Сейчас всё
«работает» по везению: предикаты ссылаются на пакетную копию, а проверки
регистрации (`cls in spec.entities`) проходят, потому что зарегистрированы
*обе* копии. Любая будущая проверка, опирающаяся на identity (дубликаты
имён, подсчёт сущностей, reachability по типам), сломается. У pytest та же
история болезни — годы багов двойного импорта и спецрежим `importmode=importlib`
как лекарство.

Вывод: интуиция верна — кастомный механизм не просто оверхед, он **некорректен**
для multi-file случая, ради которого и делался.

## Три архитектуры загрузки

### A. Сейчас: обход файлов + getmembers

Лоадер сам находит `.py`, грузит каждый в обход пакетной системы, сканирует
модули. Плюс один: работает без точки входа. Минусы: двойной импорт (баг
выше), хаки с `sys.path` и именами модулей, своя логика обнаружения,
непредсказуемость для IDE и отладчика.

### B. Точка входа + стандартный импорт

Спека — обычная Python-программа. `spec.py` импортирует остальное; analint
делает **один** `importlib.import_module` (или вообще получает уже
импортированный модуль), дальше Python сам строит граф импортов — один раз,
с правильным кешированием в `sys.modules`, без дубликатов.

Важное наблюдение: **пользователь уже пишет эти импорты.** `rules.py` уже
импортирует `entities`, `scenarios.py` — `use_cases`. Единственное, чего не
хватает, — чтобы `spec.py` импортировал «верхушки» графа (scenarios, flows),
1–3 строки. Никакой новой работы для автора — просто перестаём дублировать
работу Python.

### C. Реестр с авторегистрацией

Стандартный паттерн Django models / SQLAlchemy declarative / pytest fixtures:
базовый класс регистрирует наследников при создании
(`EntityMeta.__new__` — уже есть! одна строка), экземпляры Action/Scenario
регистрируются в `__init__`. После исполнения точки входа реестр полон —
`inspect.getmembers` не нужен вообще. Авто-сборка Spec (фича, которую мы
недавно сделали и полюбили) **сохраняется и упрощается**: Spec при проверке
просто снимает снапшот реестра.

Тонкость: глобальный реестр загрязняется временными объектами (тесты,
REPL). Решение стандартное — скоупинг: контекст загрузки
(`with registry.scope(): import_module(...)`) или снапшот-дельта. У Django
и pytest эти грабли пройдены, рецепты известны.

## Рекомендация: B + C, кастомная машинерия умирает

```
analint check path/                # ищет spec.py как точку входа — UX не меняется
analint check path/spec.py         # или явная точка входа
```

1. Лоадер = `importlib.import_module` точки входа со штатным `sys.path`-
   контекстом. ~20 строк вместо текущей машинерии.
2. Сборка = реестр на метаклассе/`__init__` со скоупом на время загрузки.
   `collect_from_modules` и `inspect.getmembers` удаляются.
3. Дисциплина «что не импортировано — того нет в спеке» становится фичей:
   граф импортов и есть явное определение состава спеки. Дешёвый страховочный
   warning: «в директории есть `.py`-файлы, не достижимые из точки входа» —
   ловит забытые файлы (сравнение множества файлов с `sys.modules`).

Что выигрываем помимо корректности: спека становится обычной Python-программой —
работает отладчик, IDE-навигация, `python spec.py` для смоук-теста,
спеку можно ставить как пакет и импортировать из других спек. Это же
подготовка к Rust-ядру (см. ROADMAP): чем тоньше и стандартнее слой загрузки,
тем чище граница «Python-фронтенд → IR → движок».

Цена: в директории без `spec.py` нужно его создать (3 строки) — приемлемо;
и миграция тестов лоадера. Дубликат-баг сам по себе оправдывает изменение.
