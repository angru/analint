# analint — Концепция

## Проблема

Бизнес-аналитику сегодня описывают в Word, Confluence, Miro. Это текст и схемы — их нельзя
проверить на противоречия, нельзя сделать diff, нельзя передать агенту. С другой стороны,
код — это слишком детально и технически, бизнес его не читает.

analint занимает промежуточное место: **Python-код, читаемый как требования, верифицируемый
как спецификация**.

---

## Чего не хватает в текущей модели

Сейчас: _snapshot verification_ — берём снимок мира (`given`) и проверяем, что правила
выполняются. Нет времени, нет последовательности, нет изменений состояния.

Реальный бизнес-процесс — это цепочка:
1. **Что было до** (предыдущие use cases завершились, накоплено состояние)
2. **Что происходит** (use case выполняется, меняет состояние)
3. **Что стало после** (новое состояние, возможно запустились другие use cases)

---

## Расширенная модель концептов

### 1. Entity + StateMachine

Сущность — это не просто набор полей. Она живёт во времени и переходит между состояниями.
StateMachine описывает, какой use case вызывает какой переход:

```
Order: PENDING → PAID    (via: Checkout)
Order: PAID   → SHIPPED  (via: Ship)
Order: PENDING → CANCELLED (via: Cancel)
```

Линтер проверяет: нет ли переходов вне схемы, нет ли "висящих" состояний без выхода.

---

### 2. BusinessRule — три разных типа

| Тип | Когда | Пример |
|---|---|---|
| **Invariant** | Всегда | `Product.price > 0` — никогда не может быть отрицательным |
| **Precondition** | Перед UC | `Wallet.balance >= Order.total` — до Checkout |
| **Postcondition** | После UC | `Order.status == PAID` — после успешного Checkout |

Это принципиально разные вещи. Смешивать их в одну кучу — терять смысл.

---

### 3. UseCase — полная картина

UseCase — это не просто "набор правил". Это описание **что происходит**:

```
UseCase: Checkout
  Actor:         Customer
  Requires:      [Login]          ← предыдущие UC, которые должны завершиться
  Preconditions: [sufficient_funds, stock_available, order_pending]
  Effects:       Order.status → PAID
                 Wallet.balance -= Order.total
  Emits:         OrderPlaced(order_id, total, customer_id)
  Postconditions:[order_is_paid, wallet_debited]
```

`Requires` — это зависимость по времени: Checkout можно запустить только если Login уже был.
`Effects` — декларативное описание изменений, которые происходят.
`Emits` — события, которые запускают другие use cases.

---

### 4. Event — соединение без жёсткой связи

Вместо прямой цепочки `UC_A → UC_B` — события:

```
Checkout    emits: OrderPlaced
Payment     triggered_by: OrderPlaced
Notification triggered_by: OrderPlaced
```

Это ближе к реальности: бизнес не знает, кто обрабатывает событие. Линтер проверяет,
что каждое эмитированное событие подхвачено хотя бы одним UC, и что поля события
совпадают с тем, что ожидает подписчик.

---

### 5. Flow — пользовательский путь

Flow — это последовательность use cases, описывающая полный пользовательский сценарий,
с ветвлениями:

```
Flow: HappyCheckout
  Login
  BrowseCatalog
  AddToCart
  Checkout ──→ [if stock available]   → Payment → Confirmation
              [if out of stock]       → OutOfStockNotice
              [if insufficient funds] → PaymentFailed
```

Линтер проверяет: постусловия каждого шага логически совместимы с предусловиями следующего.

---

### 6. Scenario — начальное состояние + поток + финальные утверждения

Сценарий задаёт начальное состояние, выполняет flow (или отдельный UC), утверждает итог:

```
Scenario: successful_checkout
  Given:  Order(total=50, status=PENDING)
          Wallet(balance=100)
          Product(stock=5, price=50)
  When:   Flow(HappyCheckout)   ← или конкретный UseCase
  Then:   Order.status == PAID
          Wallet.balance == 50   (100 - 50)
          Event(OrderPlaced) was emitted
  Expected: pass
```

Это отвечает на вопрос: "Если мир выглядел так, и пользователь сделал вот это —
получим ли мы ожидаемый результат?"

## Что линтер проверяет

### Структурные проверки (статические)
- Все ссылки на поля сущностей существуют
- Все UC в Flow зарегистрированы
- Все события, которые `emits`, подхвачены хотя бы одним `triggered_by`
- StateMachine покрывает все возможные состояния entity
- Нет циклических зависимостей в `requires` без явного описания цикла

### Логические проверки (против данных сценария)
- Инварианты выполняются для всех данных во всех сценариях
- Предусловия UC выполняются на момент его вызова в flow
- Постусловия выполняются после применения effects
- Ожидаемые финальные состояния совпадают с результатами

### Предупреждения
- UC описан, но не входит ни в один flow
- Состояние entity в given не достижимо по StateMachine (откуда оно взялось?)
- Postcondition одного UC противоречит precondition следующего в flow

## Принципы дизайна DSL

**Декларативность:** описываем *что*, не *как*. Нет алгоритмов, нет циклов, нет if-else в коде.

**Выразительность:** `Wallet.balance >= Order.total` читается как требование, не как код.

**Минимум строк:** id/name — строки по необходимости. Статусы — enum. Ссылки на поля — через класс.

**Compositional:** flow из UC, UC из rules, rules из предикатов над полями.

**Верифицируемость:** каждый концепт в DSL либо проверяется статически (ссылки),
либо проверяется против данных (предикаты), либо документируется явно (строки).

---

## Что НЕ входит в scope

- **Реализация логики** — analint описывает *что должно быть*, не *как это реализовать*
- **Оркестрация** — не запускает use cases, не вызывает API
- **Рантайм-мониторинг** — это статический + сценарный анализ
- **UI/визуализация** — JSON-вывод для внешних инструментов, не рендеринг схем

## Связь с существующими инструментами

| Инструмент | Что делает | Что делает analint иначе |
|---|---|---|
| BPMN | Схемы процессов | Code as spec, git-trackable, machine-verifiable |
| TLA+ / Alloy | Формальная верификация | Менее формально, ближе к бизнес-языку |
| OpenAPI | Контракты API | Бизнес-уровень, не HTTP-уровень |
| Gherkin/BDD | Сценарии на натуральном языке | Типизированные сущности, проверяемые инварианты |
| Event Storming | Воркшоп для описания событий | Постоянный артефакт, не одноразовое мероприятие |

analint ближе всего к **Gherkin + типизация + формальные инварианты**.
