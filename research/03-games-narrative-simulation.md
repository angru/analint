# Игры, сюжет, симуляции: примерка DSL к доменам

## 1. Сюжет как планировочный домен

Согласованность сюжета — это формально задача о достижимости в графе
состояний. Маппинг на концепты analint почти дословный:

| Нарративное понятие | Концепт analint | Статус |
|---|---|---|
| Факт мира («ключ у героя», «NPC мёртв») | поле Entity | ✅ есть |
| Сцена / сюжетный бит | UseCase (pre + effects) | ✅ есть |
| «Эту сцену нельзя сыграть, пока…» | PRECONDITION | ✅ есть |
| Арка персонажа (жив → ранен → мёртв) | StateMachine | ✅ есть |
| Чеховское ружьё (введено → должно выстрелить) | Event: emitted → triggered_by | ✅ есть (warning уже работает!) |
| Порядок глав | Flow + requires | ✅ есть |
| Выбор игрока (ветвление) | — | ❌ нет недетерминизма |
| Несколько NPC / предметов инвентаря | — | ❌ синглтоны |
| «Финал достижим из любой развилки» | — | ❌ нет reachability |
| «Игрок не может застрять» (dead-end) | — | ❌ нет reachability |

Верхняя половина таблицы — приятный сюрприз: существующий warning
«event emitted but never triggers a use_case» — это буквально детектор
несработавших чеховских ружей. DSL уже говорит на языке нарративного
дизайна, не зная об этом.

### Мини-пример: квест на текущем DSL

Что выразимо уже сейчас:

```python
class Hero(Entity):
    has_sword: bool = False
    has_key: bool = False
    location: Location = Location.VILLAGE

class Dragon(Entity):
    alive: bool = True

class Princess(Entity):
    rescued: bool = False

uc_take_sword = UseCase(
    id="take-sword",
    rules=[BusinessRule(id="at-smith", rule_type=RuleType.PRECONDITION,
                        expression=Hero.location == Location.SMITHY)],
    effects=[Set(Hero.has_sword, True)],
)

uc_slay_dragon = UseCase(
    id="slay-dragon",
    rules=[
        BusinessRule(id="armed", rule_type=RuleType.PRECONDITION,
                     expression=Hero.has_sword == True),
        BusinessRule(id="dragon-alive", rule_type=RuleType.PRECONDITION,
                     expression=Dragon.alive == True),
    ],
    effects=[Set(Dragon.alive, False)],
)

uc_rescue = UseCase(
    id="rescue-princess",
    rules=[BusinessRule(id="dragon-dead", rule_type=RuleType.PRECONDITION,
                        expression=Dragon.alive == False)],
    effects=[Set(Princess.rescued, True)],
    requires=[uc_slay_dragon],
)
```

Работает! Сценарий «герой без меча пытается убить дракона» с
`Expect.FAIL` — уже валидная проверка сюжетной дыры. Но дальше упираемся:

**Вопрос, который хочется задать:** «существует ли вообще последовательность
сцен от начала к `Princess.rescued == True`?» И обратный: «есть ли состояние,
из которого финал недостижим?» (классика: игрок продал меч до дракона —
softlock). Текущий движок на это не отвечает — нужно, чтобы автор сам
придумал ломающий сценарий. **Но softlock — это по определению ситуация,
которую автор не предусмотрел.** Ручные сценарии тут принципиально не работают.

### Что нужно сюжетному профилю (по убыванию)

1. **Reachability-запросы** — `Reachable(Princess.rescued == True)`,
   `NoDeadEnd(goal=...)`, `Inevitable(...)`. Это ядро ценности.
2. **Choice / недетерминизм** — `uc_confront = UseCase(..., outcomes=[spare, kill])`.
   Без ветвления интерактивный нарратив не описать.
3. **Несколько экземпляров** — NPC, предметы. Можно жить на синглтонах
   удивительно долго (каждый важный NPC — свой класс), но инвентарь и
   «любой стражник» требуют множественности.
4. **Создание/удаление** — персонажи появляются и умирают, предметы
   расходуются.

Важно: пункты 1–2 дают рабочий «линтер сюжета» **даже на синглтонах**.
Прототип нарративного профиля не требует решать проблему множественности.

## 2. Геймплейные системы (не сюжет)

Крафт, экономика, прогрессия — тоже pre/effects-системы, и проверки той же
природы: «можно ли скрафтить топ-предмет?», «нет ли дюпа ресурсов?»
(инвариант сохранения), «нет ли тупика прогрессии?». Дополнительно к
сюжетным нуждам здесь критичны:

- **числа и арифметика** — стоимости, урон, ресурсы. Пространство состояний
  становится бесконечным → нужны границы (`balance: int [0..1000]`) или
  абстракция. Решаемо, но дороже enum'ов.
- **агрегаты** — `Sum(Item.weight) <= Hero.capacity`.

Баланс (а не согласованность) — отдельный жанр (см. Machinations в
02-related-work.md), туда идти не обязательно.

## 3. Симуляции

Честная оценка: **полноценные симуляции — другой продукт.** Им нужны время,
планировщик событий, стохастика, метрики по прогонам. Это конфликтует с
идентичностью «линтер/верификатор».

Но есть дешёвый промежуточный режим: **исполнение спецификации** —
случайные/исчерпывающие блуждания по уже описанным действиям
(`analint simulate --steps 100`). Это не симуляция мира, а smoke-test
спецификации: находит состояния-ловушки и мёртвые действия без формальных
запросов. Ceptre показала, что один формализм выдерживает оба режима
(анализ + исполнение). Дёшево, потому что движок reachability и движок
блуждания — один и тот же код с разной стратегией обхода.

## 4. Описание систем «как было изначально» (микросервисы и т.п.)

Системная аналитика (v1.0 из роадмапа: Service, Topic, Queue) ложится на ту
же базу: сервис — Entity, взаимодействие — Action, топология — инварианты.
Reachability здесь тоже главный приз: «событие, опубликованное A, доходит
до B», «нет цикла синхронных вызовов». Т.е. **движок один на все три
домена** — аргумент в пользу инвестиций именно в движок, а не в доменные фичи.

## Вывод по доменам

| Домен | Пригодность ядра | Главный недостающий кусок |
|---|---|---|
| Бизнес-аналитика (текущий) | ✅ работает | агрегаты, множественность |
| Сюжет / нарратив | ✅ маппинг почти 1:1 | reachability + choice |
| Геймплейные системы | ⚠️ да, но | числа с границами, агрегаты |
| Симуляции | ⚠️ только как «исполнение спеки» | время, стохастика — не брать |
| Системная архитектура | ✅ ложится | reachability по топологии |

Во всех строках, где ядро пригодно, недостающий кусок один и тот же —
**движок поиска по состояниям**. Это и есть ответ на вопрос «надо ли
переделывать DSL»: не DSL надо переделывать.
