# Реформа уровня поля: Lifecycle на сущности, Field-констрейнты, типизация

Пять замечаний к API после стресс-теста; все приняты. Решения и обоснования.

## 1. Lifecycle привязывается от сущности, а не наоборот

**Проблема.** `Lifecycle(field=Card.status, ...)` живёт отдельной переменной,
на которую никто не ссылается («мёртвая ссылка» — работает только через
неявную сборку), а описание поля размазано: тип и дефолт — в сущности,
жизненный цикл — где-то ещё. Декларативность хочет полного описания поля
в одном месте.

**Решение.** Lifecycle — это *значение по умолчанию* поля:

```python
class Card(Entity):
    status: CardStatus = Lifecycle(
        initial=CardStatus.TODO,
        transitions=[
            Transition(CardStatus.TODO, [CardStatus.IN_PROGRESS, CardStatus.ARCHIVED]),
            ...
        ],
        terminal=[CardStatus.ARCHIVED],
    )
```

Метакласс при создании класса: дефолт поля = `initial`, обратные ссылки
(`entity_cls`, `field_name`) инжектятся автоматически (пользователь больше
не пишет `field=` — циклическая ссылка устранена), id = `"Card.status"`.
Спека собирает жизненные циклы из классов сущностей — модульные переменные
не нужны вовсе. Паттерн каноничный: так конфигурируют поля Django, SQLAlchemy
и pydantic.

## 2. Констрейнты уровня поля: `Field(...)`

**Проблема.** Однопольные ограничения жили в трёх местах: мировой
`Invariant(Warehouse.stock >= 0)`, движковый `Bounds(Warehouse.stock, 0, 2)`,
и нигде — при конструировании инстансов.

**Решение.** Одно объявление на поле, по образцу pydantic:

```python
class Warehouse(Entity):
    stock: int = Field(0, ge=0, le=2)

class Message(Entity):
    disturbances: int = Field(0, ge=0, le=2, saturate=True)  # счётчик-порог
```

Одна запись даёт сразу три механизма:
1. **валидацию при конструировании** — `Warehouse(stock=-1)` падает сразу,
   плохой `given` не доживает до раннера;
2. **проверку пост-состояния в раннере** — эффект, выводящий поле за
   границы, даёт находку;
3. **границы для движка** — то, что делал `Bounds` (твёрдая граница =
   находка + отсечение; `saturate=True` = клампинг). Класс `Bounds` удалён.

**Правило размещения ограничений** (ответ на «инвариант это или нет»):
ограничение **одного поля** (диапазон, не-пусто) → на поле через `Field`;
ограничение, **связывающее поля или сущности** (`Membership.board_id ==
Board.id`, `Implies(delivered, captured)`) → мировой `Invariant`. Мир —
для отношений, поле — для домена значений.

## 3. StrEnum в примерах

`requires-python >= 3.11`, так что `StrEnum` доступен. Фреймворку безразлично
(любой Enum работает), но StrEnum даёт бесплатную сериализацию в JSON/IR и
читаемые значения. Примеры переведены; в доках — рекомендация, не требование.

## 4. Типизация вместо `object`

`object` как type hint — это «не написали тип». Введены:
- базовые классы **`Predicate`** (наследуют `_Eq`, `_And`, …) и **`Effect`**
  (наследуют `Set`, `Subtract`, `Add`) — теперь сигнатуры честные:
  `pre: list[Predicate]`, `effect: list[Effect]`, `evaluate(pred: Predicate, ...)`;
- **`Generic[S]`** для `Lifecycle[S]` / `Transition[S]` — initial, переходы
  и терминальные состояния одного типа;
- `Any` остаётся только там, где значение действительно произвольно
  (значения полей) — это честный `Any`, а не ленивый `object`.

Полная mypy-строгость — отдельная задача в roadmap (дорожка экосистемы).

## 5. `Transition.to_states` — всегда коллекция

Нормализация «скаляр или список» в `__post_init__` противоречила имени
(множественное число) и удивляла. Теперь: `to_states` — всегда коллекция
(хранится неизменяемым кортежем), скаляр — `TypeError` с подсказкой.
`Transition(PAID, [CANCELLED])` — единственный способ записать один переход.
