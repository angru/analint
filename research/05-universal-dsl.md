# Универсальный DSL: набор примитивов и эргономика

Задача: убрать аналитическую специфику (UseCase, BusinessRule) и определить
минимальный универсальный набор примитивов — одинаково естественный для
аналитика, гейм-дизайнера и системного архитектора. Плюс: сделать DSL
читаемым «обычным человеком», а не похожим на математическую нотацию.

## 1. Диагноз: откуда многословность

Посчитаем поверхность текущего API: `Entity, Actor, Event, BusinessRule,
RuleType, UseCase, Scenario, Expect, StateMachine, Transition, Set, Subtract,
Add, Assert, Emitted, Flow, Spec` + 6 комбинаторов предикатов = **23 имени**.
Для сравнения: ink держится на ~2 концептах (knot + divert), Gherkin — на 3
(Given/When/Then). Успех обоих — именно в крошечном словаре.

Главный источник шума — **BusinessRule как объект-обёртка**. Одно логическое
утверждение стоит 5–6 строк церемонии:

```python
rule_card_not_archived = BusinessRule(
    id="card-not-archived",
    name="Card must not be already archived",
    rule_type=RuleType.PRECONDITION,
    expression=Card.status != CardStatus.ARCHIVED,
)
# ...а потом ещё надо не забыть включить её в uc.rules=[...]
```

Хотя само утверждение — одна строка: `Card.status != CardStatus.ARCHIVED`.
Причём `rule_type=PRECONDITION` дублирует информацию, которая и так следует
из места употребления: прекондиция — это то, что лежит в `pre` действия.

Второй источник — **обязательные id и name**, которые почти всегда повторяют
имя переменной. Лоадер уже ходит по модулям через `inspect.getmembers` и
*знает* имя переменной — id можно выводить автоматически.

## 2. Предлагаемое ядро: 8 примитивов

| # | Примитив | Было | Роль | Почему это имя |
|---|---|---|---|---|
| 1 | `Entity` | Entity | типизированное состояние | уже универсально |
| 2 | `Invariant` | BusinessRule(INVARIANT) | ограничение мира, верно всегда | общее слово трёх доменов |
| 3 | `Action` | UseCase | переход: pre → do → post | нейтрально + совпадает с planning |
| 4 | `Event` | Event | сигнал между действиями | уже универсально |
| 5 | `Actor` | Actor | кто совершает действия | театр + UML, всем понятно |
| 6 | `Lifecycle` | StateMachine | допустимые состояния поля | «жизненный цикл» понятен дизайнеру, «конечный автомат» — нет |
| 7 | `Scenario` | Scenario | конкретный пример: given/then | Gherkin-наследие, узнаваемо |
| 8 | `Spec` | Spec | корень + метаданные | в профилях — алиасы (`World`) |

**Ключевое структурное изменение:** pre/post-условия перестают быть
объектами и становятся **голыми предикатами по месту употребления**:

```python
archive_card = Action(
    by=Member,
    pre=[
        Card.status != CardStatus.ARCHIVED,
        Card.board_id == Board.id,
    ],
    effect=[
        Set(Card.status, CardStatus.ARCHIVED),
        Subtract(Board.card_count, 1),
    ],
    emits=[CardArchived(card_id=Card.id)],
)
```

`effect` — существительное намеренно: это не команды, а **факты о следующем
состоянии** (одновременные, неупорядоченные, правые части — на пред-состоянии;
см. 07-declarative-semantics.md). `emits` принимает шаблон события с биндингом
полей к выражениям над состоянием — поля Event перестают быть мёртвым грузом.

Что исчезло: `BusinessRule`, `RuleType`, `id=`, `name=`, `entities=[...]`
(выводится из упомянутых полей — движок и так их видит), `rules=[...]`.
Что осталось содержательного: **всё**.

`RuleType` уходит целиком: INVARIANT → `Invariant(...)` на уровне модуля,
PRECONDITION → `pre=`, POSTCONDITION → `post=` (опциональная двойная
бухгалтерия: проверка, что `do` сделал то, что обещано).

Переиспользуемые правила никуда не деваются — это просто переменные:

```python
board_is_active = Board.status == BoardStatus.ACTIVE   # переиспользуемый предикат

create_card  = Action(pre=[board_is_active, ...], ...)
archive_card = Action(pre=[board_is_active, ...], ...)
```

Именование для отчётов: лоадер берёт имя переменной (`board_is_active` →
"board is active" в сообщении об ошибке). Опциональный `label="..."` — для
тех, кто хочет человеческий текст.

### Слои поверх ядра

```
Слой 1 (ядро, 90% случаев):  Entity, Action, Invariant, Scenario, Spec
Слой 2 (жизненные циклы):    Lifecycle, Transition, Set/Add/Subtract
Слой 3 (связность):          Actor, Event, Flow, requires
Слой 4 (движок, будущее):    Reachable, NoDeadEnd, AlwaysHolds, реляционные эффекты (f.next)
```

### Доменные профили — алиасы, не форки

```python
# analint.narrative
World = Spec;  Scene = Action;  Character = Actor;  Fact = Invariant
# analint.systems
System = Spec;  Operation = Action;  Service = Actor
```

Один движок, одна семантика; профиль — модуль с переименованиями и
доменными пресетами проверок. Гейм-дизайнер читает `Scene(pre=..., do=...)`
и не подозревает про Hoare-тройки.

## 3. Принципы читаемости («не математика»)

Сформулированы по итогам кейс-стади (06) и сравнения с ink/Gherkin/TLA+:

1. **Тест чтения вслух.** Каждая конструкция должна читаться предложением:
   `Action(pre=[Hero.has_sword == True], effect=[Set(Troll.alive, False)])` →
   «действие: если у героя есть меч — тролль перестаёт быть живым». TLA+
   этот тест проваливает (`[]<>(...)`) — и это главная причина его
   непринятия, при всей мощи.
2. **Слова вместо символов** для всего, кроме сравнений, которые знают все
   (`==, >, >=`). Никаких `<<`, `|=>`, кастомных операторов. `Set/Add/Subtract`
   остаются словами. Добавить `Implies(a, b)` — «если a, то b» сейчас
   приходится писать как `Or(Not(a), b)`, что нечитаемо (находка кейс-стади).
3. **Ноль обязательной церемонии.** id/name/description — опциональны,
   выводятся из имён переменных. Каждое обязательное поле — налог на каждую
   строчку спеки.
4. **Одно утверждение — одна строка.** Списки плоских предикатов вместо
   вложенных формул. `pre=[a, b, c]` вместо `expression=And(a, And(b, c))`.
5. **Место употребления = семантика.** Не спрашивать у пользователя
   `rule_type`, если он уже положил предикат в `pre`.
6. **Ошибки — на языке домена и с историей.** Не «rule failed», а трасса:
   «купив два зелья, герой остаётся без меча — тролля не победить, мост
   недостижим». Контрпример-трасса читается как мини-сюжет — это фича
   именно нарративного движка.

## 4. Сравнение размера словаря

| | концептов | церемония на 1 правило |
|---|---|---|
| analint сейчас | 23 | ~6 строк (BusinessRule + регистрация) |
| предлагаемое ядро | 8 (+5 комбинаторов) | 1 строка (предикат в `pre=`) |
| ink | ~2 | — (но и проверок нет) |
| Gherkin | 3 | — (нет семантики, только текст) |
| PDDL | ~6 | низкая, но LISP-синтаксис |
| TLA+ | ~10 | высокая + математическая нотация |

Целевая позиция: словарь уровня PDDL, читаемость уровня Gherkin,
проверки уровня bounded model checking.

## 5. Таблица миграции

| Сейчас | Станет |
|---|---|
| `UseCase(id=, name=, rules=[BusinessRule(PRECONDITION, expr)], effects=)` | `Action(pre=[expr], effect=[...])` |
| `emits=[OrderPlaced]` (класс-метка) | `emits=[OrderPlaced(total=Order.total)]` (шаблон с биндингом) |
| `triggered_by=[OrderPlaced]` | `on=OrderPlaced` (+ pre над payload события) |
| `BusinessRule(rule_type=INVARIANT, expression=e)` | `Invariant(e)` |
| `BusinessRule(rule_type=POSTCONDITION, expression=e)` | `Action(..., post=[e])` |
| `RuleType` | — (исчезает) |
| `StateMachine` | `Lifecycle` |
| `uc.entities=[...]` | — (выводится из предикатов и эффектов) |
| `id="..."` обязательный | опциональный, из имени переменной |
| `Or(Not(a), b)` | `Implies(a, b)` |

Семантика рантайма (4-фазный прогон сценария, контекст, resolve/evaluate)
**не меняется вообще** — это переупаковка поверхности API, не переписывание.
Старые имена могут жить как deprecated-алиасы один релиз.

## 6. Открытые вопросы

- ~~**`do=` vs `effects=`**~~ — **решено в пользу `effect=`**
  (07-declarative-semantics.md): `do` — глагол, провоцирует императивное
  прочтение; эффекты — это факты о следующем состоянии, не команды.
  Семантика — одновременная (правые части на пред-состоянии), порядок в
  списке не значим, два эффекта на одно поле — ошибка.
- **Параметризация действий** (`move(card, to_column)`) — нужна вместе с
  множественностью экземпляров (этап 3 из 04-direction.md), не раньше.
  На синглтонах действия-без-параметров честно работают (06 это подтверждает).
- **Derived-поля** (`Bar.dark` вычисляется из `Player.has_cloak`) — кейс-стади
  показало нужду; кандидат: `dark = Computed(Player.has_cloak == True)`.
  Отложить до движка: derived-поля в snapshot-режиме можно заменить инлайном
  предиката.
