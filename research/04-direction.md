# Вердикт и направление

## Вердикт

**Переделывать DSL с нуля не нужно.** Тройка «Entity-состояние + правила-предикаты
+ действия с pre/effects» — это независимо переоткрытый планировочный домен,
проверенный десятилетиями в planning и model checking. Концепты универсальны
для всех целевых доменов (аналитика, сюжет, геймплей, системная архитектура).

**Узкое место — движок, а не язык.** Snapshot-проверка отвечает только на
вопросы, которые автор догадался задать. Все «вкусные» гарантии — достижимость
финала, отсутствие softlock'ов, инварианты на *всех* достижимых состояниях —
требуют поиска по пространству состояний. Хорошая новость: над enum-полями
это пространство конечно, и bounded BFS — инженерная задача, не научная.

## Что делать с концептами

### UseCase → Action: переименовать, семантику оставить

«UseCase» — словарь бизнес-аналитики; «Action» нейтрален и точен для всех
доменов (и совпадает с термином в planning). Семантика pre + effects —
правильная, не трогать. `Actor`, `requires`, `emits/triggered_by` — тоже
универсальны (в нарративе: персонаж, порядок сцен, чеховские ружья).

### Ядро + профили вместо одного словаря

Ответ на «выглядит сложновато»: сейчас всё экспортируется одним плоским
списком из 14 имён, и непонятно, что обязательно, а что — продвинутое.
Разделить:

```
ядро (хватает для 90% случаев):  Entity, Rule, Action, Scenario, Spec
слой 2 (жизненные циклы):        StateMachine, Transition, Effects
слой 3 (связность):              Actor, Event, Flow, requires
профили (словари-синонимы):      analint.narrative → Scene = Action, Character = Actor…
```

Профиль — это не форк семантики, а тонкий слой имён + доменные пресеты
проверок. Дёшево, а порог входа для гейм-дизайнера снижает радикально.

## Roadmap-эскиз (по убыванию value/cost)

### Этап 1 — Reachability engine (главный приз)

Состояние = кортеж значений полей всех сущностей. BFS из `initial` по всем
действиям, чьи preconditions выполнены. Новые проверки:

```python
Reachable(Princess.rescued == True)        # цель достижима
NoDeadEnd(goal=Princess.rescued == True)   # из любого достижимого состояния цель ещё достижима
AlwaysHolds(Wallet.balance >= 0)           # инвариант на всех достижимых состояниях, не только в сценариях
DeadActions()                              # действия, чьи preconditions недостижимы
```

При нарушении — выдавать **трассу-контрпример** (последовательность действий) —
это качественно лучшее сообщение об ошибке, чем «rule failed».
Числовые поля на этом этапе: либо требовать явные границы, либо исключать
из поиска (проверять только в сценариях, как сейчас). Честное ограничение.

Это превращает analint из «test runner для спекуляций» в верификатор,
и одновременно открывает нарративный и системный домены.

### Этап 2 — Choice (недетерминизм)

`outcomes=[...]` у Action: несколько альтернативных наборов effects.
Runner проверяет каждый исход; reachability ветвится. Без этого нет
интерактивного нарратива; для бизнес-домена тоже полезно (успех/отказ платежа).

### Этап 3 — Множественность и кванторы

Самый дорогой пункт: ломает `Context = dict[type, Entity]`, требует
параметризованных действий и `ForAll/Exists/Count`. Путь Alloy: bounded
scope («проверяй для ≤3 экземпляров»). Делать только когда этапы 1–2
докажут спрос — на синглтонах нарративный прототип уже жизнеспособен.

### Этап 4 — Дешёвые расширения по запросу

- guards на Transition (statecharts-lite)
- условные эффекты `Set(..., when=pred)`
- `analint simulate` — случайные блуждания по действиям (smoke-test спеки)
- иерархические состояния — только если попросят

## Чего НЕ делать

- **Не строить TLA+** — полная темпоральная логика, liveness, fairness:
  мощно, но убивает доступность, которая и есть преимущество analint.
  Фиксированный набор именованных запросов (`Reachable`, `NoDeadEnd`,
  `AlwaysHolds`) покрывает практические нужды без LTL-нотации.
- **Не идти в количественные симуляции** — время, стохастика, распределения
  (Machinations-территория). Другой продукт, другой движок, другой пользователь.
- **Не добавлять выразительность авансом** — каждый новый примитив до
  появления движка увеличивает «сложновато», не добавляя гарантий.
  Правило: сначала движок умеет проверять, потом язык умеет выражать.

## Риски

| Риск | Оценка | Митигация |
|---|---|---|
| State explosion при поиске | реальный, но отложенный | enum-поля + bounded scope; для синглтонов пространства крошечные |
| Числовые поля делают поиск бесконечным | реальный | явные границы или исключение из поиска (graceful degradation до текущего поведения) |
| DSL станет ещё сложнее | главный продуктовый риск | ядро/профили; новые проверки — это новые *функции*, а не новые концепты |
| «Между двух стульев»: аналитикам — слишком формально, формалистам — слишком игрушечно | стратегический | целиться в пустую нишу (см. 02): люди, которым TLA+ дорого, а Confluence мало; нарратив — там вообще никого |

## Ответы на исходные вопросы одной строкой

- **Хватает ли выразительности?** Для снапшот-проверок — да; для обещанных
  доменов — нет, но дыра в движке, а не в языке.
- **Перерабатывать ли DSL?** Нет. Переименовать UseCase → Action, разрезать
  на ядро/профили, дальше наращивать движок.
- **UseCase — удачная концепция?** Да, это STRIPS-оператор; неудачно только имя.
- **Сосредоточиться на состояниях/процессах/ограничениях?** На **поиске по
  состояниям**. Ограничения и процессы уже есть; не хватает того, кто по ним ходит.
- **Игры/сюжет — реалистично?** Сюжет — да, маппинг почти 1:1, ниша пустая;
  нужны reachability + choice. Симуляции — только в форме «исполнить спеку».
