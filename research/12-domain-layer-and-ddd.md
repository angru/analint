# Доменный слой, DDD и место analint в терминологии

Вопрос: «бизнес-слой» и «доменный слой» — синонимы? Как analint соотносится
с Domain-Driven Design? И главное практически: сага — это домен или уже
системная аналитика (сервисы)?

## 1. Бизнес-слой vs доменный слой

Практически — синонимы. «Бизнес-логика» / «бизнес-слой» — старший термин
(из трёхзвенной архитектуры 90-х: presentation / business / data).
«Доменный слой» — термин DDD (Эрик Эванс, 2003), уточняющий то же самое:
domain — это предметная область, которой служит софт; доменный слой — код,
выражающий её правила в чистом виде, без транспорта и хранения.

Уточнение, которое даёт DDD и которое нам полезно — **четыре слоя**:

```
presentation   — UI, API-контракты            ← не наша зона
application    — оркестровка use-case'ов,     ← граница (см. ниже)
                 транзакции, координация
domain         — сущности, правила, события,  ← ЗОНА ANALINT
                 процессы предметной области
infrastructure — БД, очереди, HTTP, сервисы   ← не наша зона (= «системная
                                                 аналитика», отложена)
```

analint описывает **доменный слой** (и кусочек application — порядок и
условия выполнения операций). Дальше используем слово «домен»; «бизнес-слой»
— допустимый синоним в разговоре.

## 2. analint — это исполняемый тактический DDD

Сопоставление словарей даёт почти построчное совпадение:

| DDD (тактические паттерны) | analint | Комментарий |
|---|---|---|
| Entity (сущность с идентичностью) | `Entity` | то же слово, тот же смысл |
| Invariant (правило согласованности агрегата) | `Invariant` | Эванс использует именно это слово |
| Domain Event | `Event` + `emits`/`on` | каноничный domain event с payload |
| Command / Application Service | `Action` (pre/effect/post) | контракт команды |
| Aggregate (граница согласованности) | контекст действия (синглтоны given) | не формализован, но фактически есть |
| Value Object | enum'ы, значения полей | частично; нет first-class VO |
| Saga / Process Manager | процесс = Lifecycle + Actions + Events | см. §3 |
| Ubiquitous Language | сама спека | спека и есть единый язык, но исполняемый |

И стратегические паттерны ложатся на наш роадмап:

| DDD (стратегические) | analint | Статус |
|---|---|---|
| Bounded Context | композиция спек / поддомены | пробел из research/10 §2.2 — это и есть bounded contexts! |
| Context Map | `Service`-маппинг + события между спеками | research/11, отложено |
| Anti-corruption layer | — | вне зоны |

Вывод по терминологии: **analint честно описывается как «исполняемый
тактический DDD»** — спецификация доменного слоя, которую можно проверить.
Это не маркетинговая натяжка, а наблюдение о совпадении словарей, возникшем
независимо (мы пришли к Entity/Invariant/Event/Action из соображений
универсальности — см. research/05 — и попали в лексикон Эванса почти
дословно). Практическое следствие: в документации можно смело опираться
на DDD-термины — это готовый общий язык с аудиторией бэкенд-разработчиков.

## 3. Сага — это домен, а не сервисы

Сомнение понятное: сага ассоциируется с микросервисами, очередями,
eventual consistency — то есть с инфраструктурой. Но это смешение уровней:

- **Суть саги — доменный процесс**: заказ проходит стадии (размещён →
  зарезервирован → оплачен → отгружен), и на каждый сбой есть
  компенсация (платёж не прошёл → вернуть резерв). Утверждения вида
  «нельзя оказаться одновременно оплаченным и отменённым», «отмена после
  оплаты обязана вернуть деньги», «заказ не может зависнуть навсегда» —
  это чистый домен, ни одного слова про HTTP.
- **Микросервисы — одна из реализаций** этого процесса. Распределённость
  и очереди — инфраструктурная проекция; в доменной модели от неё остаётся
  только то, что и так важно бизнесу: **промежуточные состояния видимы**
  (заказ бывает «оплачен, но не отгружен») и **шаги могут сбоить**
  (у платежа есть исход FAILED).

Поэтому сага моделируется в текущем analint без новых концептов:

| Сага (паттерн) | Доменная модель в analint |
|---|---|
| шаг саги | `Action` со status-guard'ами в pre |
| событие между шагами | `Event` (emits/on) — доменное событие, не сообщение очереди |
| компенсация | обычный `Action` (release_reservation, refund_payment) |
| исходы шага (успех/сбой) | альтернативные действия с одинаковым pre — ветвление берёт на себя движок |
| состояние процесса | `Lifecycle` заказа с terminal-состояниями |
| «сага всегда завершается» | `NoDeadEnd(goal=Order в терминальном статусе)` |
| «не списали дважды / не зависли деньги» | `Unreachable(...)` |

Никакого `Service` в примере не будет — это осознанная проверка тезиса
«домен отделим от топологии». Привязка действий к сервисам (кто реализует
reserve_stock) — отдельный, *крупнозернистый* слой из research/11, и он
живёт поверх доменной спеки, не внутри неё.

## 4. Результаты стресс-теста (examples/fulfillment/)

Сага реализована: 5 сущностей, 16 действий (включая 5 компенсаций),
4 жизненных цикла с терминальными состояниями, 5 событий с payload,
2 инварианта, 19 сценариев, 6 запросов движка. **558 строк** на всё
(~35 строк на действие со сценариями и запросами), 8 файлов. Движку
хватило **34 состояний** — комбинаторика статусов жёстко срезается
guard'ами.

### Главная находка: движок поймал автора в первые секунды

Первая версия `supplier_restock` имела pre `stock < 2`. `NoDeadEnd` тут же
выдал тупик с трассой: `restock → reserve → decline_payment → restock →
restock` — пока платёж падал, поставщик забил полку до предела, и
компенсации некуда вернуть зарезервированный товар (выход за `Bounds`
склада) → заказ завис навсегда. Это **реальный доменный баг** («возвраты
блокированы переполненным складом»), внесённый автором ненамеренно и
найденный машиной до первого запуска сценариев руками. Лучшей демонстрации
ценности движка не придумать.

### Прочие наблюдения

- **Ветвление парами действий работает.** authorize/decline и
  delivery/lost с одинаковыми pre — движок честно исследует обе ветки.
  Отдельный примитив Choice пока не понадобился.
- **Цель-констелляция выразима.** `Or` из пяти `And`-комбинаций
  «согласованных концовок» — самый сложный предикат проекта — читается
  нормально. При его составлении пришлось *думать* (потерянная посылка —
  это CANCELLED+REFUNDED+**CONSUMED**, товар не возвращается на полку), и
  это продумывание само по себе — работа по верификации модели.
- **`on=`-цепочки задокументировали хореографию,** а исследование шло по
  status-guard'ам в pre — для саги этого достаточно, шаги и так охраняются
  статусами. Event-pool семантика не понадобилась.
- **Ограничение DeadActions:** он работает в рамках одного начального
  мира. Обе ветки склада (есть товар / нет товара) удалось оживить в одном
  мире только введением `supplier_restock`. В общем случае нужен union по
  нескольким начальным состояниям — кандидат в улучшения движка.
- **Сервисы не понадобились нигде** — тезис §3 подтверждён: сага целиком
  доменная, распределённость остаётся проекцией реализации.

### Вердикт стресс-теста

Реальный многошаговый процесс с компенсациями выражается целиком, на
текущих примитивах, в ~560 строк, и верифицируется за миллисекунды.
Граница выразительности по-прежнему — множественность (несколько
одновременных заказов, конкуренция за остатки — невыразимо), но
одиночная сага — зона уверенного владения.
