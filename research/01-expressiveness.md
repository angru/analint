# Выразительность текущего DSL: формальная характеристика

Чтобы понять, на что DSL способен, полезно описать его в терминах формальных
моделей, а не фич. Тогда пределы видны сразу.

## Что такое analint-спецификация формально

### Состояние

Состояние мира — это **фиксированный кортеж синглтонов**: ровно один экземпляр
на каждый тип сущности.

```python
Context = dict[type, Entity]   # rule_checker.py:9
```

Поля — скаляры: enum, число, строка, None. Никаких коллекций, ссылок на другие
экземпляры (есть `board_id: str`, но это просто строка — «join» вида
`Card.board_id == Board.id` работает только потому, что `Board` один).

**Формально:** состояние = присваивание значений конечному множеству
переменных `EntityType.field`. Это **propositional state**, как в SAT/STRIPS,
а не реляционная структура, как в Alloy или базе данных.

### Правила

Предикаты — это **бескванторная пропозициональная логика** над сравнениями
полей: `=, ≠, <, ≤, >, ≥, In, IsNull` + `And/Or/Not`.

Чего нет:

- **Кванторов** — нельзя сказать «у каждой карточки есть колонка» или
  «существует участник с ролью OWNER». При синглтонах кванторы и не нужны —
  это связанные ограничения.
- **Арифметики в выражениях** — нельзя написать
  `Wallet.balance - Order.total >= 0` или `Board.card_count == Count(Card)`.
  Только `поле ⊙ поле` или `поле ⊙ литерал`.
- **Агрегатов** — Count, Sum, Max отсутствуют. `Board.card_count` в taskboard —
  ручная денормализация именно из-за этого, и линтер не может проверить,
  что счётчик согласован с фактическим числом карточек.

### Действия (UseCase)

`UseCase` = **guarded command**: охрана (preconditions ∧ invariants) +
параллельное присваивание (effects: Set / Add / Subtract).

Это в точности **STRIPS-оператор** из автоматического планирования:
precondition + add/delete effects, только над переменными со значениями
вместо булевых фактов (т.е. ближе к SAS+ — финитно-доменному планированию).
Важный и, видимо, ненамеренный результат: DSL уже является языком описания
**планировочного домена**.

Чего нет в действиях:

- **Условных эффектов** — «если карточка была последней, заархивировать
  колонку» не выразить.
- **Создания/удаления сущностей** — effects только мутируют существующие
  синглтоны. «Создать уведомление» невозможно — в taskboard это обходится
  тем, что Notification уже лежит в `given`.
- **Недетерминизма** — у действия ровно один исход. Выбор («игрок согласился
  или отказался»), случайность, ошибки внешних систем — невыразимы.
- **Параметров** — действие не принимает аргументов. `uc_move_card` двигает
  «карточку вообще», а не конкретную карточку в конкретную колонку. При
  синглтонах это не мешает; при множественных экземплярах станет блокером.

### Модель верификации

Сценарий — это **проверка тройки Хоара на одной точке**:

```
{given}  UseCase  {postconditions ∧ then}
```

Пользователь руками задаёт состояние, движок делает **один шаг** и проверяет.
`Flow` проверяет только порядок (`requires` против позиции в списке), но
**не прогоняет состояние через цепочку** — эффекты шага N не становятся
входом шага N+1.

`StateMachine.reachable_states()` — единственный кусочек настоящего
state-space анализа, и он работает над абстракцией «одно enum-поле»,
игнорируя всё остальное состояние.

## Сводная таблица пределов

| Свойство | Есть? | Что блокирует |
|---|---|---|
| Снапшот-проверка правил | да | — |
| Один шаг действия (pre → effects → post) | да | — |
| Несколько экземпляров одного типа | **нет** | `Context = dict[type, Entity]` |
| Кванторы (ForAll / Exists), агрегаты | **нет** | предикаты + синглтоны |
| Арифметические выражения | **нет** | resolve() понимает только поле/литерал |
| Условные эффекты | **нет** | effects — плоский список присваиваний |
| Создание/удаление сущностей | **нет** | effects мутируют существующее |
| Недетерминизм / выбор | **нет** | один исход на действие |
| Цепочка шагов с протаскиванием состояния | **нет** | Flow проверяет только порядок |
| Достижимость состояния (reachability) | почти нет | только BFS по одному enum-полю в SM |
| Темпоральные свойства («когда-нибудь», «всегда после») | **нет** | нет трасс вообще |
| Время, длительности, параллелизм | **нет** | нет модели времени |

## Ключевой вывод

Текущий analint — это **STRIPS-домен с ручными тест-кейсами**. Язык описывает
переходы, но движок не ищет — он только проверяет предъявленное. Поэтому
гарантии существенно слабее, чем кажется: «16 PASS» означает «16 придуманных
автором ситуаций согласованы», а не «система согласована».

Хорошая новость: раз состояние — присваивание enum-полей (числа — отдельная
проблема), пространство состояний **конечно**. Bounded model checking /
исчерпывающий BFS над ним — реализуемая задача, не исследовательская.
Это главный кандидат на апгрейд (см. 04-direction.md).
