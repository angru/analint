# Недетерминированная инициализация: отдельный пробел, не Choice действий

Повод: перевод
[Quint play_mafia.qnt](https://github.com/quint-co/quint/blob/main/examples/games/mafia_werewolf/play_mafia.qnt)
в `examples/mafia/spec.py`.

В Quint `init` выбирает роль каждого игрока недетерминированно. В analint
перевод пришлось зафиксировать на одной расстановке: Mahtab — Mafia,
Gabriela и Max — Citizens. `Param` сохранил недетерминированный выбор
участников действий, но не выбор начального мира.

## Точный диагноз

В analint уже есть недетерминизм переходов:

- если в состоянии разрешены несколько `Action`, explorer ветвится по всем;
- `Param` разворачивает семейство действий во все конечные bindings;
- будущие реляционные эффекты смогут дать несколько post-state одному
  действию.

Но начальное состояние всегда одно:

```text
defaults сущностей + query.given -> один context -> один state key -> BFS
```

То есть отсутствует не «недетерминизм вообще», а **множество допустимых
начальных состояний** (initial-state relation / initial state set).

Это ортогонально Choice действий:

```text
I(s)             какие состояния могут быть начальными
Next(s, s')      какие переходы возможны после старта
```

`Param` и несколько enabled actions расширяют `Next`. Они ничего не меняют
в `I`.

## Насколько это важно

### Не блокер для фиксированной доменной модели

Если система действительно имеет один известный старт — новый заказ всегда
`DRAFT`, игра всегда начинается в Foyer, saga всегда начинается без списания —
текущие defaults являются точной моделью. Добавление initial-choice только
расширит язык без новой гарантии.

### Существенный пробел для конфигурируемых и неизвестных стартов

Проблема становится важной, когда проверяемое свойство должно выполняться
при **любой допустимой конфигурации**:

- скрытое назначение ролей в Mafia;
- разные владельцы, лимиты и feature flags при запуске workflow;
- восстановление процесса из нескольких допустимых persisted states;
- произвольное начальное распределение ресурсов;
- проверка компонента при всех входах ограниченного домена.

Сегодня `AlwaysHolds(p)` означает:

> `p` истинно во всех состояниях, достижимых из одного конкретного snapshot.

Без явного ограничения в документации это легко прочитать сильнее:

> `p` истинно при любой допустимой инициализации модели.

Вторая формулировка сейчас неверна. Это потенциальный false-green: автор
проверил одну удобную расстановку ролей или конфигурацию и решил, что проверил
правило системы.

В Mafia различие наблюдаемо, а не теоретическое. При зафиксированном старте
с одной Mafia, двумя Citizens и первой Night граждане не могут победить:
Mafia сразу убирает одного Citizen, после чего два живых игрока голосуют друг
против друга и получают ничью. Другие назначения ролей и ограничения на
число Mafia относятся к исходной Quint-модели, но текущий analint-query их
не перебирает.

## Почему обходные пути недостаточны

### Несколько query с разными `given`

Это честный способ проверить конечный список конфигураций вручную, но:

- каждый query всё равно знает только один root;
- все свойства приходится дублировать для каждого root;
- легко добавить новую допустимую конфигурацию и забыть часть свойств;
- общий verdict «для всех легальных стартов» не является частью модели.

`given` остаётся полезным для локального запроса и контрпримера, но не заменяет
initial-state set.

### Setup-actions из искусственного состояния

Можно добавить `UNINITIALIZED`, затем действия `choose_mahtab_as_mafia`,
`choose_gabriela_as_mafia` и так далее. Формально BFS получит нужные ветки,
но цена высока:

- техническое состояние попадает в доменную модель и трассы;
- каждое обычное действие требует guard «инициализация завершена»;
- инварианты приходится ослаблять для setup-фазы;
- `DeadActions` и `NoDeadEnd` начинают учитывать технический протокол;
- setup становится наблюдаемым поведением, хотя это множество корней.

Это допустимая компиляция внутреннего IR, но плохая пользовательская
абстракция.

### `Param` на setup-action

Он уменьшает дублирование bindings, но не устраняет перечисленные проблемы.
Кроме того, коррелированные условия вроде «ровно один игрок — Mafia» нельзя
выразить независимым выбором значения каждого поля без общего initial
constraint.

## Нужная семантика

Explorer должен принимать конечное множество корней:

```text
initial states I = {s0, s1, ...}
BFS starts with every distinct state in I
```

Дальнейшая семантика запросов естественна:

| Query | Квантификация по initial states |
|---|---|
| `Reachable(p)` | существует root и путь к `p` |
| `Unreachable(p)` | ни из одного root нет пути к `p` |
| `AlwaysHolds(p)` | `p` во всех состояниях из всех roots |
| `NoDeadEnd(goal=p)` | из каждого достижимого состояния каждого root цель ещё достижима |
| `DeadActions()` | действие мёртвое, если не разрешено ни в одном общем графе |

Инварианты проверяются на каждом root до expansion. Трасса должна указывать
не только действия, но и исходную конфигурацию, иначе контрпример неоднозначен.

Технически это небольшое расширение текущего BFS:

- `Exploration.parents` уже допускает root с `(None, None)`;
- очередь и `states` можно заполнить несколькими state keys;
- совпавшие графы автоматически сольются по существующему `state_key`;
- cache key должен включать множество roots, а не один key.

Главная работа находится не в BFS, а в честной поверхности DSL и генерации
конечных initial states.

## Какой API не стоит выбирать сразу

Не следует добавлять императивный `nondet()` или исполняемый Python-generator.
Это снова сделает host language скрытой частью семантики и осложнит будущий IR.

Также недостаточно дать каждому полю независимый `OneOf(...)`: декартово
произведение удобно, но не выражает корреляции:

- ровно один игрок имеет роль Mafia;
- сумма начальных балансов равна фиксированному supply;
- status и payload согласованы.

Нужны две ступени:

1. **Явный конечный набор initial snapshots.** Минимальная, сериализуемая и
   понятная возможность. Покрывает Mafia и восстановление из известных
   конфигураций.
2. **Декларативное initial relation.** Домены полей + общие predicates,
   которые фильтруют кандидатов. Имеет смысл после bounded multiplicity и
   кванторов; до них correlations над фиксированными singleton entities уже
   можно писать через `And/Or/Implies`.

В реализованном прототипе выбран явный узел `Initial(vary=..., where=...)`,
который передаётся запросу через `initial=`. Он компилируется в тот же набор
корней, что и `given_any`, до запуска BFS.

## Приоритет

Это не важнее multiplicity для общих real-world моделей, но важнее
`Computed(...)`, профилей-алиасов и косметических расширений. Причины:

1. расширяет не только выразительность, но и силу существующих verification
   claims;
2. реализация multi-source BFS мала относительно bounded multiplicity;
3. два независимых перевода Quint уже указали на этот предел: coin отметил
   его частично, Mafia сделал наблюдаемым;
4. без него сравнение `AlwaysHolds` с model-checking tools требует постоянной
   оговорки «из одного выбранного начального состояния».

Рекомендация: поставить **finite initial-state sets** в v1.2+ перед
multiplicity, провести прототип на Mafia и не смешивать эту задачу с
реляционными effects/Choice.

## Результаты прототипа (реализовано)

Ступень 1 реализована как `given_any=[[...], [...]]` на запросах +
multi-root BFS в движке:

- explorer засевает очередь всеми корнями; одинаковые корни сливаются
  через существующий `state_key`; cache key — кортеж корневых ключей;
- трасса идентифицирует конфигурацию: `init #1 ⊢ mafia_kills(...) → …`,
  плюс INFO-находка с полным описанием корня;
- `given` остался сахаром для единственного корня; `given= + given_any=`
  вместе — ошибка.

Мафия переписана на роль-универсальные Param-действия (killer/victim/
bystander с ролевыми охранами в pre вместо захардкоженной Махтаб-мафии),
и все три запроса квантифицируются по трём расстановкам ролей
(36 состояний = 12 × 3). Головная теорема стала настоящей:

> `citizens_cannot_win` — Unreachable при КАЖДОЙ расстановке ролей,
> а не при одной удобной.

Синергия с Param оказалась ключевой: без роль-универсальных действий
multi-root бесполезен (действия, именующие мафию, не работают для других
корней).

Ступень 2 теперь тоже реализована:

```python
Initial(
    vary=[Mahtab.role, Gabriela.role, Max.role],
    where=[exactly_one_mafia],
)
```

- `vary` принимает конкретные поля и `BoundField` над Scope;
- конечные домены выводятся из bool/Enum, bounded integer `Field`,
  lifecycle или `Field(values=[...])`;
- `where` фильтрует полное декартово произведение обычными predicate AST;
- превышение `max_candidates`, пустая relation и evaluation errors дают FAIL,
  а не усечение или ложный PASS;
- Mafia больше не строит три root snapshot хост-языковым list comprehension:
  8 комбинаций ролей декларативно фильтруются до 3, итоговый граф остаётся
  прежним — 36 состояний.
