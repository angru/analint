# analint как инструмент для AI-агентов

Вопрос: есть ли ценность в analint именно для AI-агентов — как в инструменте,
которым агент (а) изучает систему перед изменением, (б) прорабатывает
гипотезы («что будет, если»), (в) проверяет согласованность после изменения
модели и только потом меняет код. И почему такого никто не сделал — может,
агентам это не нужно?

Дисклеймер о методе: часть этого файла — самонаблюдение агента, который
ежедневно работает с чужими кодовыми базами. Это нетипичный, но релевантный
источник данных.

---

## 1. Ландшафт 2026: идея витает в воздухе, но звено отсутствует

Проверка «никто не сделал» даёт неожиданный результат: **направление не
просто существует — оно взорвалось**, но все упёрлись в одно и то же
недостающее звено.

**Spec-Driven Development (SDD)** стал мейнстримом 2025–2026: GitHub Spec Kit
(>90k звёзд, 30+ интеграций с агентами), AWS Kiro (IDE со спеками как
first-class сущностью), Tessl («спека — источник истины, код генерируется»).
Родилось это как ответ на провал vibe coding: агенты пишут правдоподобный
код, который дрейфует от намерения и разваливается с ростом проекта.
Но во **всех** этих инструментах спека — это markdown на естественном языке
(requirements.md, EARS-нотация). Её нельзя проверить на противоречивость,
по ней нельзя задать запрос «достижимо ли состояние X», и она гниёт так же,
как Confluence — только теперь её гноят ещё и агенты.

**Формальные методы + LLM** — горячая академическая тема: PAT-Agent
(автоформализация для model checking), VeriGuard (верифицированная генерация
кода), роадмапы «LLM + formal methods для trustworthy agents», верификация
планов агентов через LTL и Kripke-структуры. Но это код-уровневая или
исследовательская тяжесть (Dafny, TLA+, PAT) — не инструмент, который
команда продукта поставит в CI завтра.

**Microsoft (опыт DSL-Copilot)** прямо формулирует уроки: LLM конфабулируют
даже при наличии грамматики, «валидация обязательна», и если у DSL есть
схема/типы — **выставляйте их агенту через MCP**.

Итого карта:

```
                       спека проверяемая?
                       нет              да
для людей      Confluence/Word     TLA+/Alloy (порог входа)
для агентов    Spec Kit/Kiro/Tessl ← ПУСТО →  (академия: PAT-Agent и т.п.)
```

Ниша «лёгкая, исполняемая, доменная (бизнес-уровень, не код-уровень)
спецификация, которую агент может запрашивать и проверять» — **пуста, при
подтверждённом спросе с обеих сторон**: SDD-волна доказала, что агентам
нужны спеки; FM+LLM-волна доказала, что агентам нужны внешние верификаторы.
analint — ровно соединение этих двух волн.

Ответ на «почему никто не сделал»: делали половинки. NL-спеки — потому что
дёшево; формальную верификацию — потому что строго. Середину не делал никто,
потому что до агентов у неё **не сходилась экономика** (см. §3).

## 2. Почему агентам это нужно: пять механизмов

### 2.1. Экономика контекста

Агент не может загрузить кодовую базу из 50 сервисов в контекст. Ему нужна
**сжатая, запрашиваемая модель мира**. Спека analint — это и есть она:
taskboard (7 сущностей, 8 действий, 16 сценариев) занимает ~400 строк и
полностью описывает доменное поведение, на восстановление которого из кода
ушли бы десятки чтений файлов и greps — с риском ошибиться. Самонаблюдение:
войдя в этот репозиторий, я понял домен taskboard из спеки за одно чтение;
эквивалентная реконструкция из реализации стоила бы на порядок больше
токенов и дала бы худшую точность.

### 2.2. LLM не умеют самоверификацию — и это доказано

Ключевой результат исследований планирования (Kambhampati et al.,
LLM-Modulo): LLM **не могут надёжно проверять собственные рассуждения** о
согласованности и планах; рабочая архитектура — LLM-генератор + внешний
*звуковой* (sound) верификатор в цикле. Агент, спрашивающий сам себя
«не сломает ли это инварианты?», получает правдоподобный, а не верный ответ.
analint — ровно тот внешний верификатор, доменного уровня. Это не «удобство»,
это закрытие фундаментальной дыры архитектуры агентов.

### 2.3. Агенты уже голосуют за такие инструменты — поведением

Эмпирический факт: агенты охотно и постоянно дёргают быстрые
детерминированные проверки — tsc, ruff, mypy, pytest — потому что это
дешёвый ground truth. Самонаблюдение: я запускаю линтер/тайпчекер после
почти каждой правки; это лучший сигнал из доступных. Но весь этот слой
проверяет **код-уровень** (типы, синтаксис, юнит-поведение). Для
домен-уровня («заказ не может быть оплачен дважды», «удаление доски не
оставляет карточек-сирот») инструмента в этом ряду нет — есть только
интеграционные тесты: медленные, флаки, требующие окружения. analint
запускается за миллисекунды, без окружения, детерминирован — он встаёт
в ряд `tsc`/`ruff`, но этажом выше. Инструменты из этого ряда агенты
*используют сами*, без напоминаний — достаточно строчки в CLAUDE.md.

### 2.4. Трасса-контрпример — идеальный формат ошибки для LLM

«Тест упал» заставляет агента дебажить. Трасса
`buy_potion → buy_potion → цель недостижима` — это готовый репро и
объяснение причины в одном артефакте, в терминах домена, а не стектрейса.
Для LLM это качественно лучший вход: короткий, семантичный,
самодостаточный. (Связь с 04: движок reachability нужен агентскому
сценарию ещё сильнее, чем человеческому.)

### 2.5. Спека как постоянная память

Агенты теряют контекст между сессиями; знания о домене переносятся через
CLAUDE.md/AGENTS.md — прозу, которая никем не проверяется и дрейфует.
Спека analint — это **проверяемая** версия той же памяти: следующая сессия
агента наследует не заметки, а верифицируемую модель. Это решает реальную
боль мультисессионной и мультиагентной работы.

## 3. Главный тезис: агенты инвертируют экономику, убившую идею

Почему люди десятилетиями не поддерживают формальные модели систем
(и поэтому «компании живут без этого»):

| Фактор | Для человека | Для агента |
|---|---|---|
| Стоимость поддержки спеки при каждом изменении | высокая (главный убийца идеи) | ~нулевая: правка спеки — часть того же прогона |
| Дисциплина («сначала спека») | не выдерживается, дедлайны | тривиально обязуема: CI-гейт + строка в системных правилах |
| Выгода от чтения спеки | средняя (человек умеет читать код) | огромная (см. 2.1: контекст — главный дефицит) |
| Выгода от верификации | средняя (опытный человек «чует» противоречия) | критическая (см. 2.2: агент не чует) |
| Рассинхрон спека↔код | неизбежен → доверие падает → спеку бросают | детектируем и чиним тем же агентом в том же цикле |

Та самая причина, по которой «никто это не написал» — двойная работа, —
исчезает, когда писателей двое и один из них бесплатный. **Это сильнейший
аргумент за проект в целом**: агентский сценарий не «ещё один рынок» для
analint, а среда, в которой идея впервые становится экономически
жизнеспособной.

## 4. Рабочий цикл агента (формализация задуманного)

```
0. Ориентация      agent: analint show / query      — модель мира за один вызов
1. Гипотеза        agent: правит спеку (или --what-if), запускает check/diff
                   ⟲ итерации, пока модель не согласована
2. Ревью           human: смотрит diff СПЕКИ — маленький, семантичный,
                   читаемый («стал возможен переход PAID→CANCELLED, ок?»)
3. Реализация      agent: меняет код сервисов, ориентируясь на спеку
4. Гейт            CI: analint check + (будущее) конформанс спека↔код
```

Сдвиг в пункте 2 — отдельная ценность: человек ревьюит **смысловой дифф
модели**, а не 40 файлов кода. Это ответ на главную боль надзора за
агентами: код-диффы агентов слишком велики для содержательного ревью,
спека-диффы — нет.

Идея пользователя про «помеченные невозможные состояния» формализуется
красиво: `Unreachable(Order.status == PAID & Wallet.balance < 0)` —
утверждение «система не должна уметь сюда попадать». Сегодняшний линтер
проверит его на сценариях, движок — на всём пространстве, а **семантический
дифф** двух версий спеки поймает регрессию: «изменение делает достижимым
состояние, помеченное как недостижимое» — ровно описанный сценарий
«сервисы работают, а несогласованность всплывёт потом».

## 5. Интерфейс для агентов: что выставлять

Принципы: JSON всюду (есть), стабильные id (есть), говорящие exit-коды,
короткие ответы (контекст дорог). Поверхность — CLI и/или MCP-сервер
(одно и то же ядро):

| Команда | Вопрос агента | Стоимость реализации |
|---|---|---|
| `analint check --format json` | «всё согласовано?» | **есть сейчас** |
| `analint affects Card.status` | «что я задену, меняя это поле?» — действия, правила, сценарии, SM | **дёшево сейчас**: статический кросс-референс по уже собранной модели |
| `analint show action archive-card` | «что делает X, какие у него pre/effects?» | дёшево сейчас |
| `analint check --what-if patch.py` | «проверь гипотезу, не трогая файлы» | средне |
| `analint diff <rev>` | «что изменилось в модели между версиями?» (семантически) | средне |
| `analint query reachable/trace/deadend` | «достижимо ли? как? можно ли застрять?» | движок (04) |
| `analint simulate --steps N` | «погуляй по модели, покажи неожиданное» | движок |

Важно: `affects`/`show` — это impact analysis, реализуемый **сегодня**, без
reachability-движка, и уже один он закрывает значимую часть сценария
«узнать перед изменением». Хорошая новость для приоритизации.

## 6. Честные риски

### 6.1. Спека↔код: экзистенциальный, нерешённый

analint проверяет согласованность спеки **с самой собой** — не соответствие
кода спеке. Без моста спека может гнить молча — тот же провал, что у
Confluence, только дороже. Митигации по нарастающей:

1. **Дисциплина** — CI-гейт + правило для агента «изменил поведение — измени
   спеку». Работает, но это вера, а не проверка.
2. **Якоря трассируемости** — `@implements("archive-card")` в коде сервиса;
   линтер сверяет покрытие: каждое действие спеки реализовано где-то,
   каждый якорь ссылается на существующее действие. Дёшево, грубо, полезно.
3. **Генерация конформанс-тестов** — сценарии спеки транслируются в скелеты
   интеграционных тестов; спека начинает *исполняться против кода*.
4. (Исследовательски) сверка трасс рантайма с переходами модели.

Без хотя бы уровня 2 агентский сценарий со временем теряет доверие. Это
обязательный пункт роадмапа, если идти в агентов.

### 6.2. «Может, агенты не думают в таких терминах?»

Думают. Pre/post-условия, конечные автоматы, инварианты — плотная часть
обучающего корпуса (контракты, тесты, типы, документация). Самонаблюдение:
структурированная спека для меня *проще* прозы — её не нужно
интерпретировать, по ней можно вычислять. Подтверждение из практики
Microsoft: проблема агентов с DSL — не «непонимание парадигмы», а
конфабуляция деталей синтаксиса, и лечится она валидатором + схемой через
MCP — то есть ровно тем, чем analint и является. Питоновский DSL здесь
дополнительное преимущество: Python агенты пишут лучше всего.

### 6.3. Когда это НЕ нужно (границы применимости)

Малые кодовые базы, влезающие в контекст; одноразовый код; задачи без
доменной семантики (UI-правки, рефакторинги, перфоманс); домены, где логика
и есть код (алгоритмы). Зона ценности: многосервисные системы с нетривиальными
бизнес-правилами, долгая жизнь, много рук (людей и агентов). Игры с их
гигантскими кодовыми базами и хрупкой согласованностью механик/сюжета —
внутри зоны.

### 6.4. Холодный старт

Для существующей системы спеку нужно сначала написать. Но это самообслуживаемая
проблема: первый же агент может **намайнить черновик спеки из кода**
(`analint init --from-code` как функция будущего) — извлечение enum'ов,
переходов статусов и проверок из кодовой базы — задача, которую LLM делают
хорошо, а верифицирует результат человек + сам линтер.

## 7. Вердикт

**Да — и это, вероятно, главный сценарий проекта, а не побочный.**

1. Ниша пуста при подтверждённом спросе: SDD-инструменты (Spec Kit, Kiro,
   Tessl) доказали потребность агентов в спеках, но их спеки непроверяемы;
   FM+LLM-исследования доказали потребность во внешних верификаторах, но
   их инструменты неподъёмны для продуктовых команд.
2. Агенты инвертируют экономику поддержки спеки — причину, по которой идея
   не взлетала у людей (§3).
3. Научное основание прочное: LLM нуждаются во внешнем верификаторе
   (LLM-Modulo), analint — доменный верификатор с идеальным для LLM форматом
   ошибок (трассы).
4. Два условия успеха: (а) движок запросов — `affects`/`show` сейчас,
   reachability/diff далее; (б) мост спека↔код хотя бы уровня якорей —
   иначе доверие сгниёт.

### Влияние на приоритеты (поверх 04-direction.md)

| Было (04) | Стало с учётом агентов |
|---|---|
| 1. Reachability engine | 0. **Агентская поверхность сейчас**: `affects`, `show`, exit-коды, MCP-обёртка — дёшево, не ждёт движка |
| 2. Choice | 1. Reachability + `diff` + `Unreachable` (приоритет ещё вырос: трассы и семантический дифф — ядро агентской ценности) |
| 3. Множественность | 2. Мост спека↔код (якоря трассируемости) — **новый пункт**, обязательный для агентского сценария |
| | 3. Choice, множественность — по прежней логике |
